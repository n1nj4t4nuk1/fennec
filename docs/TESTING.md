# Testing

Fennec has two test suites: **unit tests** for the domain and application layers, and **end-to-end tests** for the HTTP API.

## Testing philosophy

The project uses a two-layer testing strategy. Unit tests verify that domain services behave correctly in isolation вЂ” they use mock repositories and event buses, so they run fast (milliseconds) and don't need any infrastructure. E2E tests verify that the full HTTP stack works correctly вЂ” they start a real Actix-Web service with in-memory repositories and send real HTTP requests through it.

This combination gives high confidence with minimal infrastructure. Unit tests catch logic bugs immediately during development. E2E tests catch wiring issues (wrong routes, missing handlers, serialisation errors) before deployment. Neither requires a running database or any external service вЂ” everything runs in-process with in-memory implementations.

Tests are designed to be **deterministic and independent**. Each test creates its own data using the Object Mother pattern, and each E2E test spins up a fresh application instance with its own in-memory repository. There is no shared state between tests, so they can run in any order and in parallel without interference.

## Running tests

```bash
# All unit tests
make test/unit

# All e2e tests (no external services needed вЂ” uses in-memory repo)
make test/e2e

# Only cti_api e2e tests
make cti_api/test/e2e

# Only config_api e2e tests
make config_api/test/e2e

# Single test by name
cargo test -p kernel it_publishes_a_created_event

# Everything at once
cargo test --workspace
```

### Mocks

Mock objects simulate the behaviour of real collaborators (repositories, event buses) in a controlled way. Each mock is hand-written rather than generated by a mocking framework. This is a deliberate choice: hand-written mocks implement the same trait as the production code, so the domain service cannot tell the difference. They are simple, explicit, and easy to debug. Because each mock is a plain struct with configurable behaviour, there is no hidden magic вЂ” you can read the mock code and understand exactly what it will do.

**`SourceRepositoryMock`** вЂ” configurable via behaviour enums:

```rust
SourceRepositoryMock::that_succeeds()                              // all ops succeed, find returns None
SourceRepositoryMock::that_fails_with_already_exists()             // save returns AlreadyExists
SourceRepositoryMock::that_returns_source(source)                  // find returns the given source
SourceRepositoryMock::that_finds_nothing()                         // find returns None
SourceRepositoryMock::that_returns_source_but_update_fails(source) // update returns NotFound
SourceRepositoryMock::that_fails_on_find(msg)                      // find returns Unexpected

// Assertions
repo.saved_ids()             // Vec<Uuid> of ids that reached save()
repo.update_call_count()
repo.delete_call_count()
```

**`EventBusMock`** вЂ” records every published event:

```rust
EventBusMock::new()
EventBusMock::that_fails()          // publish returns Err

// Assertions
bus.published_event_names()         // Vec<&'static str>
```

### Object Mothers

Setting up test data can be tedious вЂ” creating a valid `Source` requires constructing six value objects in sequence, and doing this by hand in every test adds noise that obscures the test's intent. The Object Mother pattern solves this by providing factory methods that generate valid, randomised domain objects in a single call. Tests that need a generic source call `SourceMother::random()`; tests that need a specific id, status, or description can pass it explicitly while letting the mother randomise everything else.

```rust
let source = SourceMother::random();
let id     = SourceIdMother::random();
let status = SourceStatusMother::inactive();
```

Mothers live next to the types they build: `tests/libs/kernel/src/source/domain/`.

### What unit tests cover

Each use case follows a standard set of four tests that verify the most important behaviours: correct repository interaction, correct event publishing, proper error propagation, and no side effects on failure. This consistency makes it easy to know exactly what is tested for every use case and to spot missing coverage at a glance. When you add a new use case, you write the same four tests вЂ” the pattern is mechanical and predictable.

| Test | What it asserts |
|---|---|
| `it_<saves|updates|deletes>_the_source` / `it_calls_*_on_the_repository` | The correct repo method is called exactly once |
| `it_publishes_a_*_event` | The event bus receives the right event name |
| `it_returns_not_found_when_*_does_not_exist` | The service propagates `NotFound` |
| `it_does_not_publish_event_when_*_fails` | No events are published on failure |

## End-to-end tests

E2E tests live in `tests/apps/<api>/`. They start the real Actix-Web service (in-memory, no external dependencies) and send HTTP requests through it.


### Test setup

E2E tests look and feel like integration tests, but they do not touch any external infrastructure. They use the same `build_state()` function as the production app, wired with `InMemorySourceRepository` instead of a database-backed implementation. This means each test exercises the real HTTP routing, middleware, serialisation, and handler logic вЂ” the only thing that differs from production is the storage layer. Because the repository trait is the same, the application services behave identically regardless of which implementation backs them.

Every test bootstraps the full application stack with two lines:

```rust
let app = test::init_service(
    App::new().app_data(build_state()).configure(configure_routes)
).await;
```

Because `build_state()` creates a new in-memory repository on every call, tests are fully isolated вЂ” no teardown required.

### What e2e tests cover

**`cti_api`**

| Endpoint | Cases tested |
|---|---|
| `GET /health` | 200 OK |
| `POST /sources` | 201 Created, 409 Conflict on duplicate id, 400 on invalid UUID / unknown `source_type` / unknown `status` |
| `GET /sources/{id}` | 200 with body, 404 on missing id, 400 on non-UUID path |
| `PUT /sources/{id}` | 200 updated, 404 on missing id, 400 on unknown `status` |
| `DELETE /sources/{id}` | 204 No Content, 404 on missing id |

**`config_api`** mirrors the same shape against `/config` / `/config/{key}`.

## CI pipeline

The CI pipeline provides a safety net that catches failures at every level before code reaches production. Unit tests run first and fast; if they pass, the slower e2e tests run; and only after both succeed does the pipeline attempt to build release artefacts and Docker images.

Tests run in GitHub Actions on every push:

```
unit-tests (stable / beta / nightly)
    в””в”Ђв–¶ e2e-tests (per app)
            в””в”Ђв–¶ dev-build
                    в””в”Ђв–¶ build (release)
                            в””в”Ђв–¶ docker-build
```

Unit tests run in a matrix across three Rust toolchain versions to catch compatibility regressions early.

## Adding tests for a new bounded context

1. Create `tests/libs/<context>/tests.rs` and `tests/libs/<context>/src/`.
2. Add the `[[test]]` entry to `libs/<context>/Cargo.toml`:
   ```toml
   [[test]]
   name = "<context>"
   path = "../../tests/libs/<context>/tests.rs"
   ```
3. Create mocks and mothers mirroring the existing patterns under `tests/libs/kernel/src/`.
4. Write one test file per use case under `tests/libs/<context>/src/<context>/application/`.

## Adding e2e tests for a new app

1. Create `tests/apps/<api>/tests.rs` and `tests/apps/<api>/src/`.
2. Add the `[[test]]` entry to `apps/<api>/Cargo.toml`:
   ```toml
   [[test]]
   name = "<api>-e2e"
   path = "../../tests/apps/<api>/tests.rs"
   ```
3. Import `build_state` and `configure_routes` from the app's library crate.
4. Write one test file per resource group.
